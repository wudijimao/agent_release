# BioAgent 新 Next.js 功能缺失单

## 1. 文档基线与分类

- 检查日期：2026-07-23
- 主仓库基线：`67b2333`
- 范围：高保真原型已经表达、但正式 Next.js 尚未形成真实闭环的功能
- 明确排除：`packages/web` 独有功能不迁移，不进入本缺失单

缺失类型：

- **缺接口**：服务端没有满足当前产品页面所需的稳定契约。
- **接口已有，前端未接**：服务端与 shared 已具备主要契约，但 Next.js 没有正式路由或 Adapter 装配。
- **接口不可用**：路由存在，但依赖的邮件、模型、存储或外部服务目前不能完成真实结果。
- **缺需求细化**：高保真交互与现有服务端模型无法直接对应，需要产品先明确范围、权限或状态规则。
- **缺验收**：实现已存在，但真实破坏性写入或特定环境链路尚未验证。

优先级：

- `P0`：阻塞核心用户闭环或正式上线。
- `P1`：主要页面或高频能力不完整。
- `P2`：次要能力、运营能力或范围待确认页面。

## 2. P0 缺失

### AUTH-GAP-01 注册闭环

- 优先级：`P0`
- 类型：**接口已有，前端未接** + **接口不可用** + **缺需求细化**
- 当前状态：
  - 高保真 `RegisterPage` 已在共享 UI 中。
  - 服务端已有：
    - `POST /api/auth/email-verification/send`
    - `POST /api/auth/register`
  - shared 已有 `RegisterRequest` 和 `EmailVerificationSendResponse`。
  - 正式 Next.js 没有 `/register` 路由和注册 Adapter。
  - 服务端已说明验证码接口存在，但邮件目前实际发不出去。
- 缺失细节：
  - 邮件服务商、发件域名、失败重试和告警没有形成可用运行环境。
  - 需要确认注册必填姓名、实验室创建/邀请码加入的最终步骤。
  - 需要定义验证码频率、倒计时、过期、错误次数和账号枚举保护。
  - 注册成功后是直接登录、回登录页，还是继续实验室引导，需要明确。
- 完成标准：
  - 邮件真实可达。
  - Next.js 路由和 Adapter 接入。
  - 注册、验证码错误/过期、邮箱已存在、限流和成功后的会话行为均验收通过。

### AUTH-GAP-02 找回密码闭环

- 优先级：`P0`
- 类型：**接口已有，前端未接** + **接口不可用**
- 当前状态：
  - 高保真 `ForgotPasswordPage` 已在共享 UI 中。
  - 服务端已有：
    - `POST /api/auth/password-reset/send`
    - `POST /api/auth/password-reset/confirm`
  - shared 已有 `PasswordResetConfirmRequest`。
  - 正式 Next.js 没有 `/forgot-password` 路由和重置密码 Adapter。
  - 邮件验证码投递当前不可用。
- 缺失细节：
  - 发送验证码、确认新密码和完成后重新登录的正式接线。
  - 验证码过期、错误次数、重发间隔、密码强度和限流提示。
  - 防止通过错误信息枚举邮箱是否存在。
- 完成标准：真实邮箱可以收到验证码，用户可以设置新密码，并用新密码登录。

### PROJECT-GAP-01 项目文档上传、新建、详情与下载

- 优先级：`P0`
- 类型：**缺接口** + **缺需求细化**
- 当前状态：
  - 项目详情能读取服务端已有的知识摘要。
  - 正式页面保留“创建文档、导入文档、打开文档”入口。
  - 点击入口只显示“服务端暂未提供契约”，不会伪造成功。
  - `/api/projects` 现有契约主要覆盖项目、成员、知识链接和项目会话，没有直接对应高保真文档上传/详情/下载的闭环。
- 缺失细节：
  - 项目文档究竟对应 Wiki2 节点、Project Knowledge Link、对象存储文件，还是独立 Document 实体。
  - 上传流程：预签名、文件登记、解析状态、失败重试、去重和删除。
  - 新建文档的编辑器、保存草稿、发布和权限模型。
  - 文档详情路由、可预览类型、下载鉴权和失效链接。
  - 文档属于 `knowledge / experiment / data` 哪个区，以及标签来源。
- 完成标准：项目内能真实创建或导入文档、查看处理状态、打开详情并按权限下载/删除。

### CHAT-GAP-02 项目文件与项目上下文引用

- 优先级：`P0`
- 类型：**缺接口** + **缺需求细化**
- 当前状态：
  - Chat 已支持 Skill、Mira 文档和附件引用。
  - 会话和项目之间已有 `projectId` 归属。
  - 项目文档资源尚不能作为 Chat 的稳定选择数据源。
- 缺失细节：
  - 项目资源目录接口及分页、搜索和权限。
  - 引用 ID 应使用项目知识链接、知识节点还是文件 ID。
  - 项目上下文是自动全量启用、按目录选择，还是每次消息显式引用。
  - 项目成员权限变化后，已有会话引用如何失效。
  - 服务端发送契约中的稳定 `contextRef` 类型和历史恢复格式。
- 依赖：先完成 `PROJECT-GAP-01` 的资源模型。
- 完成标准：Chat 可选择真实项目资源，发送、回答和历史恢复均保留稳定项目上下文。

### RELEASE-GAP-01 生产入口、HTTPS 与切换方案

- 优先级：`P0`

  - 默认直接暴露 `3002`，不依赖 Docker Compose 或 Nginx。
- 缺失细节：
  - 正式域名、HTTPS 证书和 Cookie Secure 策略。
  - Nginx/负载均衡的最终入口及 `/api`、SSE 超时配置。
  - 新旧前端切换、灰度比例、监控指标和回滚触发条件。
  - 日志采集、容器磁盘清理、旧镜像保留数量和告警。
  - 阿里云 Docker Hub 镜像加速或企业镜像仓库。
- 完成标准：域名 HTTPS 可用，登录 Cookie 正常，具备明确灰度、监控和回滚手册。

3. 完成 HTTPS、域名、镜像源、监控和新旧前端切换方案。

5. 明确 AI 用量的用户、权限、统计口径和充值定义。
6. 再决定实验详情、序列比对助手和独立设置入口是否进入当前版本。
